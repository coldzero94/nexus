# NEXUS 협업 가이드

우리가 어떻게 일하는지에 대한 기준 문서. 여기 없는 것은 이슈로 정하고, 정해지면 이 문서에 반영한다.

## 1. 역할

**개발은 둘 다 한다.** 구현 이슈는 스프린트 플래닝에서 나눠 가지고, 아래는 각자 리드하는 영역이다.

| | 찬영 | 치완 |
|---|---|---|
| 개발 | O | O |
| 리드 영역 | 아키텍처, 데이터 파이프라인, 릴리즈·TestFlight | 프로덕트·게임 디자인, XP 밸런스, 캐릭터·세계관, 유저 리서치 |
| 이 프로젝트의 첫 유저 | 2호 | **1호 (워치 실데이터)** |

기획·밸런스·아트도 개발만큼의 지분이다 — 캐릭터가 재미없으면 코드가 완벽해도 망한다.

## 2. 도구는 GitHub 하나

| 무엇 | 어디서 | 용도 |
|---|---|---|
| 할 일·아이디어·버그 | [Issues](https://github.com/coldzero94/nexus/issues) | 모든 작업의 단위. 템플릿 골라서 작성 |
| 스프린트 | [Milestones](https://github.com/coldzero94/nexus/milestones) | S0~S4. 진행률 바로 스프린트 상태 확인 |
| 작업 현황판 | [보드](https://github.com/users/coldzero94/projects/1) | Backlog → 이번 스프린트 → 진행 중 → **확인 대기(교차 확인)** → Done |
| 기획 문서 | [docs/](../docs/) | MVP 정의, 조사 보고서, 벤치마크, 이 문서 |
| 코드 변경 | Pull Request | §4 참고 — 코드 대신 스크린샷·영상을 보면 됨 |

### 이슈 작성 규칙

1. Issues → New issue → **[기능/아이디어]** 또는 **[버그]** 템플릿 선택
2. 핵심은 세 칸: **무엇을 / 왜 / 완료 기준**(어떻게 되면 끝난 것인가)
3. 스프린트 배정이 애매하면 마일스톤은 비워두고 플래닝 때 정한다

버그 리포트는 **재현 절차 + 기대 + 실제 + 스크린샷** 네 가지가 기준이다.

## 3. 스프린트 리듬 (2주 단위)

```
첫날        매주 1회       마지막 날
플래닝 30분 → 싱크 30분 → 데모+회고 30분
              ↑ 나머지는 전부 이슈 코멘트/카톡 비동기
```

- 스프린트의 목표는 [SPRINTS.md](./SPRINTS.md)에 있고, 끝나는 날 **TestFlight로 새 빌드가 나간다**.
- "완료"의 정의: 코드가 머지된 것이 아니라 **만든 사람이 아닌 다른 한 명이 폰에서 확인하고 이슈에 코멘트를 남긴 것**. 그래서 보드에 '확인 대기' 컬럼이 따로 있다.
- 회고는 거창하지 않다: 잘된 것 / 막힌 것 / 다음에 바꿀 것, 각자 1개씩.

## 4. 브랜치와 PR

```
main (항상 동작하는 버전)
 └─ feature/healthkit-import   ← 기능 단위 브랜치
     → Pull Request(PR)         ← main에 합치기 전 리뷰 지점
     → 머지                      ← main이 한 단계 업그레이드
```

- PR에는 **스크린샷 또는 화면 녹화 필수** — 리뷰는 코드가 아니라 동작으로 한다.
- PR 제목·설명과 TestFlight 빌드 노트는 제품 언어로 쓴다 (예: "걸음 수를 읽어서 일별로 합산한다").

## 5. 스프린트별 치완 리드 영역 (개발 이슈는 플래닝에서 별도 배분)

| 스프린트 | 포커스 | 산출물 |
|---|---|---|
| S0 | 기획 백로그 + XP 밸런스 설계 | 아이디어 이슈, XP 스프레드시트 v1, 캐릭터 컨셉 |
| S1 | 실데이터 QA | 워치 데이터 vs 앱 대조 결과, 버그 이슈 |
| S2 | 밸런스 튜닝 | 스프레드시트 v2 (실데이터 반영) |
| S3 | 아트 파이프라인 | 표정·장비 에셋 확정, 트리거 표 |
| S4 | 유저 테스트 운영 | 테스터 10명 모집·안내문·인터뷰 결과 |

## 6. 규칙 (짧음)

1. **아이디어는 카톡에 흘리지 말고 이슈로** — 카톡은 휘발되고 이슈는 남는다. 카톡으로 얘기했어도 결론은 이슈에 적기.
2. **막히면 24시간 안에 공유** — 혼자 3일 붙잡는 것이 최대 리스크.
3. **스프린트 데모는 깨지 않는다** — 사이드 프로젝트는 데모가 한 번 끊기는 순간 죽는다. 분량을 줄여서라도 데모는 한다.
4. **결정은 이슈에 기록** — "왜 그렇게 했더라?"의 답이 3개월 뒤의 우리를 구한다.
5. **환금성 보상은 어떤 단계에서도 금지** — 조사로 확정한 원칙 ([RESEARCH.md](./RESEARCH.md) 참고).
