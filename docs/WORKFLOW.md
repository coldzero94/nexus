# NEXUS 협업 가이드

우리가 어떻게 일하는지에 대한 기준 문서. 여기 없는 것은 이슈로 정하고, 정해지면 이 문서에 반영한다.

## 1. 역할

**개발은 둘 다 한다.** 구현 이슈는 스프린트 플래닝에서 나눠 가지고, 아래는 각자 리드하는 영역이다.

| | 찬영 | 치완 |
|---|---|---|
| 개발 | O | O |
| 리드 영역 | 아키텍처, 데이터 파이프라인, 릴리즈·배포(CI/Play) | 프로덕트·게임 디자인, XP 밸런스, 캐릭터·세계관, 유저 리서치 |
| 이 프로젝트의 첫 유저 | 2호 (에뮬레이터+합성 데이터) | **1호 (갤럭시 실데이터)** |
| 게이트 통과 후 | iOS 확장 리드 (core KMP 재사용) | Android 심화 + Wear OS 리드 |

기획·밸런스·아트도 개발만큼의 지분이다 — 캐릭터가 재미없으면 코드가 완벽해도 망한다.

## 2. 도구는 GitHub 하나

| 무엇 | 어디서 | 용도 |
|---|---|---|
| 할 일·아이디어·버그 | [Issues](https://github.com/coldzero94/nexus/issues) | 모든 작업의 단위. 템플릿 골라서 작성 |
| 스프린트 | [Milestones](https://github.com/coldzero94/nexus/milestones) | S0~S7(알파) + S8+(확장)·S9~S12(커뮤니티). 진행률 바로 상태 확인 |
| 작업 현황판 | [보드](https://github.com/users/coldzero94/projects/1) | Backlog → Ready → 진행 중 → **확인 대기(교차 확인)** → Done |
| 기획 문서 | [docs/](../docs/) | MVP 정의, 조사 보고서, 설계 문서, 이 문서 |
| 코드 변경 | Pull Request | §4 참고 — 코드 대신 스크린샷·영상을 보면 됨 |

### 보드 운영 규칙

- **Status(칸반)와 스프린트(마일스톤)는 별개 축이다.** Status는 작업 흐름만 표현하고, 어느 스프린트 소속인지는 마일스톤이 담당한다 — GitHub 표준 팀 플래닝 템플릿(Backlog/Ready/In progress/In review/Done) 구조.
- **모든 이슈는 Backlog에서 시작**한다. 플래닝에서 이번 스프린트 마일스톤의 이슈를 **Ready**로 올리고, 각자 잡을 때 '진행 중'으로 옮긴다.
- 자동으로 움직이는 것: PR 머지 시 `Closes #N`으로 이슈가 닫히면 → Done + 마일스톤 진행률 갱신. (보드 우측 상단 **Workflows**에서 "Item added to project → Backlog", "Item closed → Done", "Pull request merged → Done", "Auto-add to project"를 켜두면 수동 관리가 사라진다)
- 추천 뷰 구성 (`+ New view`): ① "이번 스프린트" — Filter: `milestone:"S0 · 세팅"` (스프린트 시작마다 필터만 갱신) ② "스프린트별" — Group by **Milestone**. 달성률은 [Milestones](https://github.com/coldzero94/nexus/milestones) 진행률 바.
### 이슈 작성 규칙

1. Issues → New issue → **[기능/아이디어]** 또는 **[버그]** 템플릿 선택
2. 핵심은 세 칸: **무엇을 / 왜 / 완료 기준**(어떻게 되면 끝난 것인가)
3. 스프린트 배정이 애매하면 마일스톤은 비워두고 플래닝 때 정한다

버그 리포트는 **재현 절차 + 기대 + 실제 + 스크린샷** 네 가지가 기준이다.

## 3. 스프린트 리듬 (2주 단위)

```
첫날        매주 1회       마지막 날
플래닝 30분 → 싱크 30분 → 데모+회고 30분
              ↑ 나머지는 전부 이슈 코멘트/카톡 비동기
```

- 스프린트의 목표는 [SPRINTS.md](./SPRINTS.md)에 있고, 끝나는 날 **새 빌드가 나간다** — 지금은 GitHub Release에서 APK 설치, Play 계정 후엔 내부 트랙(스토어 설치·자동 업데이트).
- "완료"의 정의: 코드가 머지된 것이 아니라 **만든 사람이 아닌 다른 한 명이 폰에서 확인하고 이슈에 코멘트를 남긴 것**. 그래서 보드에 '확인 대기' 컬럼이 따로 있다.
- 회고는 거창하지 않다: 잘된 것 / 막힌 것 / 다음에 바꿀 것, 각자 1개씩.

## 4. 티켓 → 브랜치 → PR — 한 사이클로 전부 정리되는 규칙

```
이슈 #N (티켓)                      ← 모든 작업의 시작. 이슈 없으면 먼저 만든다
 └─ feat/N-짧은-설명 브랜치          ← 이슈 번호가 브랜치 이름에
     커밋: "feat: 설명 (#N)"        ← 커밋이 이슈 타임라인에 자동 표시
     → PR: 본문에 "Closes #N"       ← 연결 지점
     → Squash 머지                  ← 이 순간 자동으로:
        ✅ 이슈 #N 닫힘 → 보드 Done → 마일스톤 진행률 갱신 → 브랜치 삭제 → APK 릴리즈
```

- **이슈 없이 코드 없다** — 작업하다 새 일이 보이면 이슈부터 만들고 브랜치를 딴다.
- **1 PR = 1 이슈**가 기본. 여러 이슈를 닫으면 `Closes #A, Closes #B`로 각각 명시.
- 브랜치 이름: `feat|fix|docs|chore / 이슈번호-슬러그` (예: `feat/9-trust-filter`).
- 머지는 **Squash만 가능**(레포 설정으로 강제) — 이슈당 커밋 1개로 main 히스토리가 깔끔해지고, 머지 시 브랜치는 자동 삭제된다.
- PR에는 **스크린샷 또는 화면 녹화 필수** — 리뷰는 코드가 아니라 동작으로 한다.
- PR 제목·설명과 릴리즈 노트는 제품 언어로 (예: "걸음 수를 읽어서 일별로 합산한다").
- **PR 제목 = squash 커밋 메시지** (레포 설정으로 강제): `type: 제품 언어 요약 (#N)` 형식 — main 히스토리가 이 제목들로만 쌓인다.
- **확인 방법 섹션이 비어 있으면 리뷰를 시작하지 않는다** (React 리포의 강제 문구 차용) — 교차 확인이 우리 '완료'의 정의라서.
- **PR 전 에이전트 리뷰(#182)**: Claude 세션에서 `/pr-review-toolkit:review-pr`을 돌려 Critical/Important 지적을 반영한 뒤 PR을 연다. 플러그인은 레포의 `.claude/settings.json`이 선언하고 있어 레포를 신뢰(trust)한 세션에는 자동 설치가 제안된다. 상대 PR 교차 리뷰도 같은 방식 — 지적은 PR 코멘트로 남긴다. 에이전트 리뷰는 사람 교차 확인(기기 검증)을 대체하지 않는다.
- 리뷰 코멘트 무게 표기(선택): ⛏ 니트픽(무시 가능) / 💭 질문 / 🛑 블로커(머지 전 해결) — 코멘트의 무게를 오독하지 않기 위한 관례.
- 자동화 전제: 보드 Workflows에서 "Item closed → Done"이 켜져 있어야 함 (§2 보드 운영 규칙).

### 커밋 메시지 규칙

**형식: `type(범위): 요약 (#이슈번호)`** — 한 번만 설정하면 자동 검사된다: `git config core.hooksPath .githooks`

- **type 7종**: `feat`(기능) · `fix`(버그) · `docs`(문서) · `refactor`(동작 불변 정리) · `test` · `chore`(설정·잡일) · `ci`
- **범위**(선택): 모노레포 모듈 — `core` `app` `health` `data` `ui` `web` `supabase` `server`
- **요약**: 한국어, 마침표 없음, 50자 안팎, **첫 줄만 봐도 무엇인지** — 히스토리는 첫 줄로 훑는다 (Google CL 가이드·Beams 공통 원칙)
- **본문**(선택): 빈 줄 뒤에 — 어떻게(how)는 코드가 말하니 **무엇을/왜**만
- **산식 버전·DB 스키마가 바뀌면 `feat!:`** — 본문에 마이그레이션/소급 재계산 경로 명시
- 좋은 예: `feat(health): 신뢰 필터 3종 — 수기 입력 제외 (#9)` / 나쁜 예: `수정`, `버그 픽스`, `wip`
- **브랜치 안에서는 작게 자주 커밋** — main에는 squash로 PR 제목 하나만 남으니 부담 없이. 단 형식을 지키면 리뷰어가 커밋 단위로 변경을 따라 읽을 수 있다

## 6. 규칙 (짧음)

1. **아이디어는 카톡에 흘리지 말고 이슈로** — 카톡은 휘발되고 이슈는 남는다. 카톡으로 얘기했어도 결론은 이슈에 적기.
2. **막히면 24시간 안에 공유** — 혼자 3일 붙잡는 것이 최대 리스크.
3. **스프린트 데모는 깨지 않는다** — 사이드 프로젝트는 데모가 한 번 끊기는 순간 죽는다. 분량을 줄여서라도 데모는 한다.
4. **결정은 이슈에 기록** — "왜 그렇게 했더라?"의 답이 3개월 뒤의 우리를 구한다.
5. **환금성 보상은 어떤 단계에서도 금지** — 조사로 확정한 원칙 ([RESEARCH.md](./RESEARCH.md) 참고).
