# NEXUS 협업 가이드 (치완용 온보딩 문서)

이 문서는 개발을 처음 접하는 사람 기준으로 쓰였다. 모르는 게 나오면 이슈에 `질문` 라벨 달고 물어보면 된다 — 그게 이 프로젝트의 정상 동작이다.

## 1. 우리 둘의 역할

| | 찬영 | 치완 |
|---|---|---|
| 정체 | 엔지니어링 | 프로덕트 오너 + 게임 디자이너 |
| 하는 일 | 아키텍처, Claude Code로 구현, PR·릴리즈, TestFlight 배포 | 기획 이슈 작성, XP·밸런스 설계(스프레드시트), 캐릭터·세계관·문구, 데모 빌드 테스트/QA, 유저 인터뷰 |
| 이 프로젝트의 첫 유저 | 2호 | **1호 (워치 실데이터 담당)** |

비개발 업무가 이 프로젝트의 절반이다. 캐릭터가 재미없으면 코드가 완벽해도 망한다.

## 2. 도구는 GitHub 하나

| 무엇 | 어디서 | 용도 |
|---|---|---|
| 할 일·아이디어·버그 | [Issues](https://github.com/coldzero94/nexus/issues) | 모든 작업의 단위. 템플릿 골라서 작성 |
| 스프린트 | [Milestones](https://github.com/coldzero94/nexus/milestones) | S0~S4. 진행률 바로 스프린트 상태 확인 |
| 작업 현황판 | Projects 보드 | Backlog → 이번 스프린트 → 진행 중 → **치완 확인 대기** → Done |
| 기획 문서 | [docs/](../docs/) | MVP 정의, 조사 보고서, 벤치마크, 이 문서 |
| 코드 변경 | Pull Request | §4 참고 — 코드 대신 스크린샷·영상을 보면 됨 |

### 이슈 쓰는 법 (치완의 주력 도구)

1. Issues → New issue → **[기능/아이디어]** 또는 **[버그]** 템플릿 선택
2. 세 칸만 채우면 된다: **무엇을 / 왜 / 완료 기준**(어떻게 되면 끝난 것인가)
3. 어느 스프린트인지 모르겠으면 마일스톤 비워두기 — 플래닝 때 같이 정함

버그 리포트는 **재현 절차**(내가 뭘 했더니) + **기대**(이래야 하는데) + **실제**(이렇게 됐다) + 스크린샷. 이 네 가지면 완벽한 리포트다.

## 3. 스프린트 리듬 (2주 단위)

```
첫날        매주 1회       마지막 날
플래닝 30분 → 싱크 30분 → 데모+회고 30분
              ↑ 나머지는 전부 이슈 코멘트/카톡 비동기
```

- 스프린트의 목표는 [SPRINTS.md](./SPRINTS.md)에 있고, 끝나는 날 **TestFlight로 새 빌드가 치완 폰에 도착**한다.
- "완료"의 정의: 코드가 머지된 것이 아니라 **치완이 폰에서 확인하고 이슈에 확인 코멘트를 남긴 것**. 그래서 보드에 '치완 확인 대기' 컬럼이 따로 있다.
- 회고는 거창하지 않다: 잘된 것 / 막힌 것 / 다음에 바꿀 것, 각자 1개씩.

## 4. 코드는 이렇게 흘러간다 (치완이 알아야 할 만큼만)

전에 물어봤던 "버전 분기 나눠서 업글하는 거야?"에 대한 답:

```
main (항상 동작하는 버전)
 └─ feature/healthkit-읽기   ← 기능 하나마다 가지를 쳐서 작업
     → Pull Request(PR)      ← "이 가지를 main에 합쳐도 될까?" 요청
     → 머지                   ← 합치면 main이 한 단계 업그레이드
```

- PR에는 **스크린샷 또는 화면 녹화가 필수**다. 치완은 코드 말고 이걸 보면 된다.
- PR 제목과 설명은 치완이 이해할 수 있는 언어로 쓴다 (예: "걸음 수를 읽어서 일별로 합산한다").
- TestFlight 빌드 노트도 마찬가지.

## 5. 치완 온보딩 커리큘럼 — 스프린트마다 개발 사이클 한 조각씩

| 스프린트 | 배우는 것 | 실제 과제 |
|---|---|---|
| S0 | 이슈로 일하는 법 | 아이디어 이슈 1개 작성 + XP 스프레드시트 |
| S1 | QA와 버그 리포트 | 자기 워치 데이터 vs 앱 대조 검증 |
| S2 | 스펙이 코드가 되는 과정 | 스프레드시트 결과와 앱 결과 대조, 밸런스 튜닝 |
| S3 | 에셋 파이프라인 | 표정·장비 에셋 소싱과 반영 확인 |
| S4 | 유저 테스트 운영 | 테스터 10명 모집·안내·인터뷰 |

## 6. 규칙 (짧음)

1. **아이디어는 카톡에 흘리지 말고 이슈로** — 카톡은 휘발되고 이슈는 남는다. 카톡으로 얘기했어도 결론은 이슈에 적기.
2. **막히면 24시간 안에 공유** — 혼자 3일 붙잡는 것이 최대 리스크.
3. **스프린트 데모는 깨지 않는다** — 사이드 프로젝트는 데모가 한 번 끊기는 순간 죽는다. 분량을 줄여서라도 데모는 한다.
4. **결정은 이슈에 기록** — "왜 그렇게 했더라?"의 답이 3개월 뒤의 우리를 구한다.
5. **환금성 보상은 어떤 단계에서도 금지** — 조사로 확정한 원칙 ([RESEARCH.md](./RESEARCH.md) 참고).
