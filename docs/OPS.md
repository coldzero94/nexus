# 크래시·안정성 운영 (E8-3, #48)

기준일 2026-07-20. 2인 팀 전제 — 온콜 없음, **주간 확인 루틴**으로 운영한다.

## 1. 수집 구성

| 채널 | 역할 | 상태 |
|---|---|---|
| **Sentry** (무료 5k 에러/월) | 미처리 크래시 수집 — 스택·기기·OS. **tracing off · PII off** | 래퍼 `app/crash/CrashReporting.kt`. DSN 미설정 시 완전 no-op |
| **Play vitals** | ANR율·크래시율의 공식 기준(스토어 노출·검색 랭킹에 반영) | Play Console → 품질 → Android vitals. 계정 개설(E8-5) 후 자동 |

- Sentry 프로젝트 생성(무료 가입) → DSN을 `kotlin/gradle.properties`(미커밋) 또는 CI 시크릿에 `nexus.sentry.dsn`으로 주입. 없으면 초기화 자체를 안 한다(동의 전 자동 수집 없음 — Crashlytics를 배제한 이유와 같은 원칙).
- SDK 직접 호출은 detekt `ForbiddenImport`가 차단 — 모든 옵션(tracing off, PII off, 스크린샷·뷰 계층 첨부 금지)은 래퍼가 강제한다. **화면에는 건강 파생 표시값이 있으므로 스크린샷 첨부는 절대 켜지 않는다.**

## 2. 주간 루틴 (금요일, 담당 교대)

1. Sentry Issues — 신규 크래시 확인: 발생 수 상위부터, **재현 경로를 이슈(버그 템플릿)로 등록**하고 보드 Backlog에.
2. Play vitals — **나쁜 동작 임계치** 확인: 사용자당 크래시율 1.09%·ANR율 0.47% 초과 시 스토어 노출 불이익. 초과 항목은 그 주 최우선 티켓.
3. 무료 티어 소진 확인(Sentry Stats) — §3 초과 대응.

초기 소규모(테스터 12명)에서는 vitals 임계 표본이 안 잡히므로 **Sentry가 1차 신호**, vitals는 공개 출시 후 주 신호로 승격.

## 3. 무료 티어 초과 시 (STACK.md §8)

1. **크래시 루프 의심 먼저** — 같은 스택이 수백 건이면 수집량 문제가 아니라 버그. 핫픽스 우선.
2. Sentry 프로젝트 설정 → Client Keys → **Rate Limit** 설정(예: 분당 5)으로 루프 방어.
3. 상시 초과면 수집 다이어트(환경 필터, 이슈별 mute) → 그래도 부족하면 vitals 단독 회귀(코드는 DSN 제거만).
