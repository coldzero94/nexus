# NEXUS 백엔드 설계

기준일 2026-07-14. 결론부터: **MVP에 서버는 없다. 하지만 서버를 전제로 한 계약은 지금 코드에 들어간다.** 이 문서는 그 계약과, 단계별 백엔드 아키텍처, 그리고 지금 배제할 것/열어둘 것을 정한다. (3개 영역 웹 검증 완료: 백업/기기 이전, 2단계 플랫폼, 로컬→서버 전환 패턴)

## 0. 단계별 그림

| 단계 | 시점 | 아키텍처 | 서버 |
|---|---|---|---|
| **1. 로컬** | MVP~알파 | HealthKit = 원본의 단일 진실 소스, 게임 상태는 SwiftData 로컬. 파생 상태의 클라우드 동기화 없음 | 없음 |
| **1.5 백업** | 알파 전 필수 | 수동 내보내기/가져오기 → (여유 시) iCloud Documents 스냅샷 | 없음 |
| **2. 하이브리드** | 게이트 통과 후 (커뮤니티) | 게임 상태는 로컬 유지, **리더보드만 서버**(제출형): 활동 요약 제출 → 서버가 산식 재계산 + 개연성 검증 | Supabase 서울 (1순위) |
| **3. 서버 권위** | 필요 시 | RewardEvent 원장을 서버로 임포트 — 리플레이를 신뢰하는 게 아니라 **검증하며 재계산** | 2단계 + 보조 서비스 |

## 1. 지금 MVP 코드에 들어가야 할 계약 ⚠️

나중에 추가하면 기존 데이터를 소급 보강할 수 없는 것들. 검증 결과 이 4가지가 로컬→서버 전환의 성패를 가른다:

1. **RewardEvent 멱등성 키** — HealthKit 샘플 UUID/syncIdentifier 기반. 없으면 아이폰·워치 이중 지급을 못 막고, 서버 전환 시 중복 이벤트를 구분할 수 없다.
2. **provenance 필드** — HKSource 번들 ID + wasUserEntered 플래그를 이벤트에 박제. 서버가 나중에 신뢰 등급을 재평가할 근거.
3. **단조 증가 시퀀스 번호** — 이벤트 전역 순서 보장(이벤트 소싱 표준).
4. **보상 취소는 수정이 아니라 보상 이벤트(compensating event) append** — HealthKit 샘플 삭제(워치 지연 동기화, 사용자 삭제) 시 기존 이벤트를 고치지 않고 취소 이벤트를 추가. 저장된 이벤트는 절대 수정하지 않는다.

추가 원칙:
- **산식은 순수 모듈 + 버전별 테스트 픽스처** (이미 NexusCore 설계) — 서버가 각 이벤트를 "당시 산식"으로 재검산할 수 있는 원장이 우리 설계의 최대 자산.
- **사용자 식별은 Sign in with Apple 전제** — MVP는 계정 없음(로컬 UUID)이지만, 나중에 SIWA로 연결 가능한 구조로. 어느 백엔드로든 이식됨.
- **소급 재계산 정책**: 하향 재계산은 표시 레벨에 반영하지 않는다(레벨다운 노출 = 이탈 요인). 기기 이전으로 원본이 없는 과거 이벤트는 기존 값 동결.

## 2. 데이터 경계 — 서버·iCloud로 가는 것 / 절대 안 가는 것

| 어떤 표면이든 실어도 되는 것 | 어떤 표면에도 싣지 않는 것 |
|---|---|
| XP 산출값, 레벨, 능력치, 산식 버전, 클래스 성향, 캐릭터 상태, 불투명 소스 해시, 기간별 활동 "요약"(2단계: 유형·시간·심박 존 요약) | 걸음 수·심박·거리 등 **원본 건강 수치**, HKSample 원본 참조, 정밀 위치 좌표(2단계 모임은 행정구역 수준만) |

근거: Apple 5.1.3(ii) "personal health information의 iCloud 저장 금지" — 원문에 파생 게임 수치의 정의가 없어 회색지대지만, 원값 배제 원칙을 지키면 승인 선례(Treeceps 등 걸음 기반 진행도 iCloud 동기화 앱들)와 같은 편에 선다. ActivityRecord(원본 수치 포함)는 로컬 전용으로 격리.

## 3. 백업·기기 이전 (9월 아이폰 교체 시즌 전 필수)

**시스템 기본 동작**: iCloud 백업·Quick Start는 앱 컨테이너(SwiftData 스토어 포함)를 자동 이전한다. 정식 출시 후 일반 사용자는 구현 없이도 보존됨. 문제는 **TestFlight 베타 앱은 복원 시 자동 재설치되지 않아 컨테이너 복원이 불확실**하다는 것 — 알파 테스터가 9월에 폰을 바꾸면 캐릭터가 날아갈 수 있다.

| 순위 | 방안 | 판정 |
|---|---|---|
| **1 (S4 필수)** | **수동 내보내기/가져오기** — 버전 필드를 가진 JSON(RewardEvent 전체 로그 + 캐릭터/스탯 스냅샷, 원값 배제)을 ShareLink로 내보내고 fileImporter로 복원 | 구현 ~1일, iCloud 미사용이라 5.1.3(ii) 노출 0. 테스터 가이드에 "새 기기로 설정만 피할 것" 병기 |
| 2 (S5) | **iCloud Documents 단일 JSON 스냅샷** — XP 지급 시점에 원자적 latest-wins 덮어쓰기. 백업 성공 여부를 UI에 노출 | 단일 기기·단일 작성자라 iCloud Documents 최대 약점(병합 충돌)이 구조적으로 없음. 스냅샷을 2단계 서버 업로드 페이로드로 재사용 가능 |
| 배제 | SwiftData+CloudKit 미러링 | `.unique` 금지가 RewardEvent 멱등성과 정면충돌 + iOS 26 동기화 중단(BAD_REQUEST) 미해결 회귀 + add-only 스키마 락인 |
| 배제 | NSUbiquitousKeyValueStore | 1MB 한도 — 증가하는 이벤트 로그와 상충 |
| 보류 | CKRecord 직접(CKSyncEngine) | 기술적 최선이나 2인 팀 MVP엔 과투자. 멀티 디바이스 요구 생기면 재검토 |

## 4. 2단계 아키텍처: "리더보드만 서버" 하이브리드

게임 상태 전체를 서버 권위로 옮기지 않는다. Strava(ML 이상치 탐지)·Sweatcoin(서버측 계산, 5~8% 거절)·Niantic(수동 입력 무시)이 증명한 패턴:

```
클라이언트                              서버 (Supabase)
─────────                              ──────────────
게임 상태·캐릭터: 로컬 유지             크루/시즌/리더보드 객체
기간별 활동 요약 생성  ──App Attest──▶  ① 어설션 검증 (요청 출처 보증)
(유형·시간·심박존 요약,                 ② 산식으로 점수 재계산 (클라 점수 안 믿음)
 원본 샘플 아님)                        ③ 개연성 검증 (인간 한계·일일 상한·wasUserEntered)
                                       ④ 이상 시 소프트 제재 (랭킹 제외, 밴 아님)
```

핵심 원칙: **클라이언트가 보낸 것은 사실(event)이 아니라 검증 대상 요청(command)이다.** App Attest는 "진짜 기기의 진짜 앱"만 보증할 뿐 가짜 건강 데이터를 못 막으므로(Apple 공식 입장, WWDC26) 서버측 재계산·개연성 검증과 반드시 병행.

### 플랫폼: 배제할 것 / 열어둘 것

| 후보 | 판정 | 근거 |
|---|---|---|
| **Supabase** | **1순위 — 열어둠** | 서울 리전(ap-northeast-2) 직접 지원 → 국외이전 요건 구조적 회피. Pro $25/월 + 100k MAU. Sign in with Apple 네이티브. Postgres라 리더보드를 RPC·머티리얼라이즈드 뷰로 정석 구현. Android 확장 시 REST 그대로 재사용. 약점: App Attest DIY(서드파티 라이브러리 존재), **리더보드에 RLS 직접 조회 금지**(10만 행에서 초 단위 지연 — security definer RPC로 우회) |
| Firebase | 2순위 — 조건부 | App Check(App Attest) 1st-party 통합이 유일한 강점. 단 **Auth가 미국에서만 처리** → 개인정보보호법 국외이전 고지 대상, 읽기 과금이 리더보드 워크로드와 상성 나쁨 |
| CloudKit public DB | **배제** | 서버 로직 전무(시즌 정산·안티치트 불가), Android SDK 없음 → 커뮤니티 데이터를 여기 쌓으면 이전 비용이 전면 재작성 |
| 자체 Vapor/Hummingbird | 2단계 시작점으로는 배제 | Vapor 4 유지보수 모드 + v5 알파 과도기. 2인 팀이 인증·백업·보안 패치까지 짊어짐. 3단계에서 BaaS 옆 보조 서비스(시즌 정산 배치)로만 재검토 |

**착수 규칙**: 게이트(알파 D14 리텐션) 통과 후, Supabase 서울 리전으로 1주 PoC 스파이크 → 확정. 그전에는 어떤 백엔드 코드도 쓰지 않는다.

## 5. 규제 연결 (상세: [RESEARCH.md §8](./RESEARCH.md#8-규제플랫폼-정책-체크리스트))

## 6. 리스크 요약

- **클라이언트 원장 위조**: 로컬 DB는 탈옥·백업 조작으로 위변조 가능 → 3단계 임포트는 반드시 "검증하며 재계산". 이상 계정 처리 정책(소급 차감 vs 사면)을 커뮤니티 오픈 전에 결정.
- **소급 재계산 UX**: 산식 변경·샘플 삭제로 XP가 하향될 수 있음 → 표시 레벨 불하향 정책(§1).
- **5.1.3(ii) 심사관 재량**: 파생 수치도 리젝 가능성 0은 아님 → 원값 배제 원칙 + 수동 내보내기(비-iCloud)를 기본 백업으로.
- **iCloud Documents 신뢰성**: 업로드 멈춤·0KB 파일 보고 있음 → 단일 파일 원자적 쓰기 + 복원 시 타임스탬프 검증 + 백업 상태 UI 노출.
