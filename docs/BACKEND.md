# NEXUS 백엔드 설계

기준일 2026-07-14. 결론부터: **MVP에 서버는 없다. 하지만 서버를 전제로 한 계약은 지금 코드에 들어간다.** 이 문서는 그 계약과, 단계별 백엔드 아키텍처, 그리고 지금 배제할 것/열어둘 것을 정한다. (3개 영역 웹 검증 완료: 백업/기기 이전, 2단계 플랫폼, 로컬→서버 전환 패턴)

## 0. 단계별 그림

| 단계 | 시점 | 아키텍처 | 서버 |
|---|---|---|---|
| **1. 로컬** | MVP~알파 | Health Connect = 원본의 단일 진실 소스, 게임 상태는 Room 로컬. 파생 상태의 클라우드 동기화 없음 | 없음 |
| **1.5 백업** | 알파 전 필수 | 수동 내보내기/가져오기 + Android 자동 백업(allowBackup) | 없음 |
| **2. 하이브리드** | 게이트 통과 후 (커뮤니티) | 게임 상태는 로컬 유지, **리더보드만 서버**(제출형): 활동 요약 제출 → 서버가 산식 재계산 + 개연성 검증 | **Supabase(DB·Auth) + 자체 Ktor 서버(Cloud Run 서울)** |
| **3. 서버 권위** | 필요 시 | RewardEvent 원장을 서버로 임포트 — 리플레이를 신뢰하는 게 아니라 **검증하며 재계산** | 2단계 + 보조 서비스 |

## 1. 지금 MVP 코드에 들어가야 할 계약 ⚠️

나중에 추가하면 기존 데이터를 소급 보강할 수 없는 것들. 검증 결과 이 4가지가 로컬→서버 전환의 성패를 가른다:

1. **RewardEvent 멱등성 키** — 건강 레코드 고유 id(Health Connect record id, iOS 확장 시 HK 샘플 UUID) 기반. 없으면 폰·워치 이중 지급을 못 막고, 서버 전환 시 중복 이벤트를 구분할 수 없다.
2. **provenance 필드** — dataOrigin 패키지명 + recordingMethod를 이벤트에 박제 (iOS 확장 시 HKSource+wasUserEntered). 서버가 나중에 신뢰 등급을 재평가할 근거.
3. **단조 증가 시퀀스 번호** — 이벤트 전역 순서 보장(이벤트 소싱 표준).
4. **보상 취소는 수정이 아니라 보상 이벤트(compensating event) append** — 레코드 삭제 감지(DeletionChange — 워치 지연 동기화, 사용자 삭제) 시 기존 이벤트를 고치지 않고 취소 이벤트를 추가. 저장된 이벤트는 절대 수정하지 않는다.

추가 원칙:
- **산식은 순수 모듈 + 버전별 테스트 픽스처** (이미 NexusCore 설계) — 서버가 각 이벤트를 "당시 산식"으로 재검산할 수 있는 원장이 우리 설계의 최대 자산.
- **사용자 식별은 소셜 로그인 전제(Google 우선, iOS 확장 시 Apple 추가)** — MVP는 계정 없음(로컬 UUID)이지만 나중에 연결 가능한 구조로. 어느 백엔드로든 이식됨.
- **소급 재계산 정책**: 하향 재계산은 표시 레벨에 반영하지 않는다(레벨다운 노출 = 이탈 요인). 기기 이전으로 원본이 없는 과거 이벤트는 기존 값 동결.

## 2. 데이터 경계 — 서버·클라우드로 가는 것 / 절대 안 가는 것

| 어떤 표면이든 실어도 되는 것 | 어떤 표면에도 싣지 않는 것 |
|---|---|
| XP 산출값, 레벨, 능력치, 산식 버전, 클래스 성향, 캐릭터 상태, 불투명 소스 해시, 기간별 활동 "요약"(2단계: 유형·시간·심박 존 요약) | 걸음 수·심박·거리 등 **원본 건강 수치**, 원본 레코드 참조, 정밀 위치 좌표(2단계 모임은 행정구역 수준만) |

근거: Play 건강 데이터 정책(광고·제3자 이전 전면 금지, "SDK가 의도치 않게 수신해도 위반") + iOS 확장 시 Apple 5.1.3(ii). 원값 배제 원칙을 지키면 양쪽 다 방어된다. ActivityRecord(원본 수치 포함)는 로컬 전용으로 격리.

## 3. 백업·기기 이전 (알파 전 필수)

> Android 전환 반영. iOS 확장 시의 백업 검증 결과(iCloud Documents 스냅샷 권고, CloudKit 미러링 배제)는 git 히스토리 `0c7b9db`의 이 문서 참고.

**시스템 기본 동작**: Android 자동 백업(`allowBackup`, 기기당 25MB)이 Room DB를 포함해 Google 계정으로 백업·기기 이전한다 — Room 원장이 25MB를 넘지 않게 유지하면(수년치 이벤트도 수 MB 수준) 일반 사용자는 구현 없이 보존됨. 단 자동 백업의 복원 시점·삼성 Smart Switch 경로는 편차가 있으므로 이것만 믿지 않는다.

| 순위 | 방안 | 판정 |
|---|---|---|
| **1 (S4 필수)** | **수동 내보내기/가져오기** — 버전 필드 JSON(RewardEvent 원장 + 캐릭터/스탯 스냅샷, 원본 건강 수치 배제)을 공유 시트로 내보내고 SAF로 복원 | 구현 ~1일, 클라우드 미경유라 정책 노출 0. 확실한 보존 경로 |
| 2 (기본) | **Android 자동 백업 활성 + 백업 규칙 검증** — Room 포함, 캐시·민감 파일 제외 규칙 | 공짜 안전망. 실기기 백업→복원 테스트 1회 필수 |
| 배제 | 원본 건강 수치의 클라우드 백업 | Play 건강 데이터 정책·개인정보 최소화 원칙 — 원장에는 XP 산출값·산식 버전만 (원본은 Health Connect가 소스 오브 트루스) |

**결정(#51, 2026-07-20)**: 원장 `idempotencyKey`(HC 레코드 UUID 참조)는 **본인 통제 백업 표면**(수동 JSON 파일·본인 Google 계정 자동 백업)에 한해 포함을 허용한다. 근거: ① UUID는 무작위 식별자로 건강 수치를 담지 않는다 ② §1 멱등성 계약이 키 원형 보존을 요구한다 — 내보내기 시점에 해시하면 복원 후 재동기화가 같은 세션을 다른 키로 이중 지급한다 ③ 두 표면 모두 사용자 본인 소유·통제로, §2의 "원본 레코드 참조 금지"가 겨냥하는 서버·제3자 표면이 아니다. §2 기준은 그대로 유지하며, 2단계 원장 서버 임포트 전에 **원장 키 해시화(불투명 소스 해시, E10-12)** 를 완료한다.

## 4. 2단계 아키텍처: "리더보드만 서버" 하이브리드

게임 상태 전체를 서버 권위로 옮기지 않는다. Strava(ML 이상치 탐지)·Sweatcoin(서버측 계산, 5~8% 거절)·Niantic(수동 입력 무시)이 증명한 패턴:

```
클라이언트 (Android)                     [Cloud Run 서울 · 무료] server/ = Ktor + core(KMP)
─────────                                ──────────────────────────────
게임 상태·캐릭터: 로컬 유지               ① Play Integrity + JWT 검증 (Supabase JWKS)
기간별 활동 요약 생성 ──HTTPS──▶          ② core 산식으로 재계산 — 앱과 같은 Kotlin 코드
(유형·시간·심박존 요약, 원본 아님)         ③ 개연성 검증 (인간 한계·상한·recordingMethod)
                                         ④ 결과를 Postgres에 기록, 이상 시 소프트 제재
리더보드 읽기: supabase-kt+RLS ──▶       [Supabase 서울 · 관리형] Postgres + Auth
```

핵심 원칙 두 가지:
1. **클라이언트가 보낸 것은 사실(event)이 아니라 검증 대상 요청(command)** — Play Integrity(iOS 확장 시 App Attest)는 "진짜 기기의 진짜 앱"만 보증하므로 서버측 재계산·개연성 검증과 반드시 병행.
2. **산식의 단일 진실**: 서버 재계산은 `server/`(Ktor)가 core(KMP)를 직접 의존해 수행 — Edge Functions(TS)로 산식을 재구현하는 이원화를 배제한다. 치완 스프레드시트 케이스 테이블이 서버 검증 테스트로도 그대로 쓰임. (Ktor 공식 풀스택 KMP 패턴, Confetti 선례)

**서버 구성 요점** (검증 2026-07): Ktor 3.5.x(CIO) + 멀티스테이지 Dockerfile → **Cloud Run asia-northeast3(서울)**, scale-to-zero — 무료 한도(월 200만 요청·18만 vCPU초)의 극히 일부만 사용. `max-instances 1~2` + 예산 알림으로 과금 폭주 차단. JVM 콜드스타트 수 초는 비실시간 제출이라 클라이언트 재시도로 흡수(필요 시 GraalVM native). 인증은 Supabase **JWKS 비대칭 키** 검증(신 `sb_secret`/`sb_publishable` 키 체계 — legacy 키는 2026년 말 폐기). DB는 Supavisor 풀러(session 5432) 경유 JDBC, 인스턴스당 풀 3~5. **서버 경로는 RLS를 우회하므로 모든 쿼리를 JWT sub로 스코핑하는 계층 강제.** Edge Functions는 불채택 — 웹훅은 pg_net이 Ktor를 직접 호출, keep-alive는 GitHub Actions cron이 Ktor의 DB 터치 헬스체크 호출.

### 플랫폼: 배제할 것 / 열어둘 것

| 후보 | 판정 | 근거 |
|---|---|---|
| **Supabase (DB·Auth만)** | **채택 — 역할 축소** | 서울 리전 → 국외이전 회피, Free 티어로 시작. Google 소셜 로그인, Postgres 리더보드(RLS 직접 조회 금지 — security definer RPC). **Edge Functions는 불채택**(산식 이원화 방지 — 계산은 Ktor로) |
| **자체 Ktor 서버 (계산·검증만)** | **채택** | core(KMP) 직접 의존 = 산식 단일 진실. 무상태 컨테이너라 운영 부담 최소. Cloud Run 서울 무료(월 200만 요청, scale-to-zero). 백업 플랜: Render free(콜드 60초)·Oracle A1(2026-06 무통보 반토막 전례) |
| Firebase | 2순위 — 조건부 | App Check(Play Integrity/App Attest) 1st-party 통합이 유일한 강점. 단 **Auth가 미국에서만 처리** → 개인정보보호법 국외이전 고지 대상, 읽기 과금이 리더보드 워크로드와 상성 나쁨 |
| CloudKit public DB | **배제** | 서버 로직 전무(시즌 정산·안티치트 불가), Android SDK 없음 → 커뮤니티 데이터를 여기 쌓으면 이전 비용이 전면 재작성 |
| 풀 자체 호스팅(DB·인증 포함) | **배제 유지** | 2인 팀이 백업·보안 패치·인증을 짊어지는 구조. 자체화는 무상태 계산 컨테이너(Ktor)까지만 |

**착수 규칙**: 게이트(알파 D14 리텐션) 통과 후, **Supabase 서울 + Cloud Run Ktor 왕복 1주 PoC 스파이크**(JWT 검증→core 재계산→Postgres 기록) → 확정. 그전에는 어떤 백엔드 코드도 쓰지 않는다. 보험: core에 js() 타깃+@JsExport로 산식을 Edge Functions에서 실행하는 경로를 별도 스파이크로 검증해 둠(호스팅 무료 티어 변동 리스크 헤지).

## 5. 규제 연결 (상세: [RESEARCH.md §8](./RESEARCH.md#8-규제플랫폼-정책-체크리스트))

2단계 서버 도입일 = 민감정보 별도 동의 UI + LBS 신고(방송미디어통신위, 소상공인 간이신고) + Play Integrity 적용일. 서버에 파생 스탯만 올리는 §2의 경계가 이 부담을 최소로 만든다.

## 6. 리스크 요약

- **클라이언트 원장 위조**: 로컬 DB는 탈옥·백업 조작으로 위변조 가능 → 3단계 임포트는 반드시 "검증하며 재계산". 이상 계정 처리 정책(소급 차감 vs 사면)을 커뮤니티 오픈 전에 결정.
- **소급 재계산 UX**: 산식 변경·샘플 삭제로 XP가 하향될 수 있음 → 표시 레벨 불하향 정책(§1).
- **정책 심사 재량**(Play 건강 정책·Apple 5.1.3): 파생 수치도 지적 가능성 0은 아님 → 원값 배제 원칙 + 수동 내보내기(클라우드 미경유)를 기본 백업으로.
- **자동 백업 복원 편차**: 제조사 이전 도구(Smart Switch)·25MB 한도·복원 시점 편차 → 수동 내보내기를 1순위로 하고, 실기기 백업→복원 테스트를 1회 수행.
